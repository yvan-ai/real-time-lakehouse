# ADR-0009 — Apache Airflow for pipeline orchestration

**Status**: Accepted · **Date**: 2026-07-07

## Context

The batch pipeline (Bronze→Silver→Gold → quality gate → lineage registration)
is chained by `scripts/run-batch.sh`: a shell loop polling `kubectl`. That
works, but there is no scheduling, no retry policy, no run history, and the
dependency graph lives in bash. The target stack (roadmap v2, pillar 4) calls
for a DAG-driven pipeline where a failed quality gate fails the run visibly.

Constraints:

- **RAM**: the `orchestration` namespace quota allows ≤ 1.5 Gi of limits —
  a full Helm-chart Airflow (webserver + scheduler + triggerer + workers +
  its own Postgres + Redis) needs several Gi and is out.
- **CPU**: the node runs close to its requests ceiling — every long-running
  pod must keep requests minimal.
- **Consistency**: Marquez already reuses the shared Postgres (ADR-0008);
  a second database-per-tool pattern is established.

Alternatives considered:

- **Dagster** — nicer local dev story and typed assets, but Airflow is the
  market standard the target stack names, has the larger operator ecosystem
  (`KubernetesPodOperator` is exactly the fit here), and the recruiter-facing
  value of this repo is demonstrating the standard tool. Trade-off accepted:
  Dagster's asset lineage would have overlapped with Marquez anyway.
- **Argo Workflows** — Kubernetes-native and lighter, but no scheduler UI /
  backfill semantics, and it would not demonstrate Airflow skills.
- **Keep the shell script** — no scheduling, no observability; rejected.

## Decision

Deploy **Airflow 2.x standalone-style**, sized for the local cluster:

1. **One pod** — scheduler + webserver as two processes in a single
   container (`LocalExecutor`), in the `orchestration` namespace.
   Requests 768 Mi / limits 1.25 Gi, inside the 1.5 Gi quota.
2. **Shared Postgres** — metadata DB is a new `airflow` database + role in
   the existing Postgres (namespace `streaming`), created by
   `airflow-db-init` (same pattern as `marquez-db-init`). Credentials come
   from `airflow.env` (gitignored, generated by `bootstrap.sh`) via the
   overlay `secretGenerator`.
3. **Work runs elsewhere** — the DAG uses `KubernetesPodOperator` to spawn
   task pods in `spark` (batch), `data-quality` (gate) and `lineage`
   (register script) namespaces, so the Airflow pod itself stays small and
   the existing per-namespace quotas keep applying. A dedicated `airflow`
   ServiceAccount gets namespace-scoped pod permissions there — no
   cluster-wide RBAC.
4. **DAGs from git** — the `airflow-dags` ConfigMap is generated by
   kustomize from `pipelines/orchestration/dags/`, so `kubectl apply -k`
   (or ArgoCD) is the deployment path for DAG changes.
5. **Fallback preserved** — `scripts/run-batch.sh` keeps working without
   Airflow; the DAG is the scheduled path, the script the manual one.

## Consequences

- `airflow dags trigger lakehouse_batch` runs batch → dbt → quality gate →
  lineage end-to-end; a red gate turns the DAG run red (the PDF's month-5
  target).
- Standalone mode means no HA and no horizontal scaling — acceptable for a
  single-node cluster, called out for prod (switch to the official Helm
  chart + KubernetesExecutor there).
- The shared Postgres failure domain now also takes Airflow metadata with
  it (already accepted for Marquez in ADR-0008).
- Task pods duplicate some pod-spec details from the Job manifests
  (image, env, mounts); the DAG file documents this and is the single
  place to keep in sync.
